# Plan: Add Port Forwarding Support for Trident Models

## Overview
Add the ability to serve patch encoder models via a network port, allowing the model server to run on one machine (potentially with GPU) while feature extraction clients run on other machines (potentially CPU-only or with different GPU resources).

## Current Architecture
- `run_batch_of_slides.py`: Main script that runs seg → coords → feat pipeline
- `trident/patch_encoder_models/load.py`: Contains encoder_factory() that loads patch encoders locally
- `trident/wsi_objects/WSI.py::extract_patch_features()`: Extracts features using locally-loaded encoder
- Patch coordinates stored in HDF5 format with metadata (patch_size, mag levels, etc.)

## Proposed Architecture

### 1. Create `load_model.py` - Model Server Script
**Location:** `/data/TRIDENT/load_model.py`

**Purpose:** Standalone script that loads a patch encoder and exposes it via a TCP/HTTP server.

**Arguments:**
- `--gpu`: GPU index to use (default: 0)
- `--port`: Port number to bind to (default: 3001)
- `--patch_encoder`: Name of patch encoder (default: conch_v15)
- `--patch_encoder_ckpt_path`: Optional path to local checkpoint
- `--host`: Host address to bind to (default: 0.0.0.0)

**Implementation:**
- Use `torchserve` or simple Flask/FastAPI HTTP server
- Load the patch encoder using existing `encoder_factory()` from `trident.patch_encoder_models.load`
- Expose endpoints:
  - `GET /health`: Health check
  - `POST /encode`: Accepts batch of images, returns features
  - `GET /model_info`: Returns model metadata (name, feature dimension, precision)

**Data Flow:**
1. Load patch encoder on specified GPU
2. Start HTTP server on specified port
3. For each request:
   - Receive batch of images (numpy arrays or tensors)
   - Apply eval transforms
   - Run through encoder
   - Return features as numpy array

### 12. Key Design Decisions

#### 12.1 HTTP vs Custom TCP
**Decision: Use HTTP/REST (Flask)**
- Easier to debug (curl, browser)
- Language agnostic
- Standard tools for monitoring, logging
- Sufficient performance for batch inference

#### 12.2 Image Encoding
**Decision: Base64 encoded PNG/JPEG**
- Text-based, easy to debug
- Standard web format
- Slightly slower than binary, but acceptable for batch processing
- Alternative: raw binary bytes (faster, harder to debug)

#### 12.3 Transform Handling
**Decision: Server handles transforms**
- Client sends raw images
- Server applies eval transforms
- Centralizes transform logic
- Ensures consistency

#### 12.4 Precision Handling
**Decision: Server handles precision, client receives numpy float32**
- Simpler client code
- Server can optimize based on GPU
- Reduces complexity

### 13. Performance Considerations

#### 13.1 Batching
- Default batch size: 64 (configurable)
- Larger batches = better GPU utilization but more memory
- Client should batch requests to minimize overhead

#### 13.2 Latency
- Network latency added per request
- Minimize by using larger batches
- Consider connection pooling for multiple requests

#### 13.3 Compression
- Optional: compress images before sending
- Tradeoff: CPU time vs network bandwidth
- Recommended for remote servers (over internet)

## Summary

This plan adds port forwarding support by:
1. Creating a standalone model server (`load_model.py`)
2. Adding a remote encoder client (`RemotePatchEncoder`)
3. Modifying the feature extraction pipeline to support both local and remote modes
4. Maintaining full backward compatibility with existing workflows

The implementation prioritizes:
- Simplicity and maintainability
- Backward compatibility
- Clear error handling
- Good performance for typical use cases
